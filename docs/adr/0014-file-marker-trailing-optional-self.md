# ADR 0014 — FILE 标记尾部 === 改为可选

**日期**: 2026-04-14  
**背景**: Claude 自发现（-self）；Vercel 生产日志显示 DeepSeek 输出 `// === FILE: /path` 缺少尾部 ` ===`，导致正则不匹配、parse_failed

## 问题描述

诊断日志：
```
foundMarkers: []
head: '// === FILE: /components/record/RecordForm.js\n...'
```

DeepSeek 输出了正确格式但省略了尾部 ` ===`。原正则 `/^\/\/ === FILE: (.+?) ===/` 要求尾部 `===` 必须存在，匹配失败，5512 字节的完整代码被丢弃。

## 根因

DeepSeek 对提示词中 `// === FILE: /path ===` 格式的遵从不稳定，偶尔省略尾部 `===`。

## 修复

**extract-code.ts**（3 处）：正则改为 `/^\/\/ === FILE: (.+?)(?:\s*===)?$/`，尾部 `===` 可选。捕获的路径加 `.trim()` 防御尾部空格。

**engineer-stream-tap.ts**：正则改为 `/\/\/ === FILE: (\/[^\s=]+)[^\n]*(\n)?/`，用 `[^\n]*` 吃掉路径到换行之间的所有内容（兼容有无 `===`）。路径加 `.trim()`。

**handler.ts**：诊断正则同步更新。

## 风险分析

| 场景 | 风险 | 防御 |
|------|------|------|
| 路径尾部空格 `// === FILE: /path.js   ` | 捕获含空格路径 | `.trim()` |
| 代码内出现 `// === FILE: /path` 注释 | 误判为文件边界 | 提示词禁止注释，极低概率 |
| tap 流式过早匹配 | `(?:\s*===)?` 在逐字符 feed 时可选组提前匹配 | tap 用 `[^\n]*` 替代，等 `\n` 到达才消费 |

## 预防措施

- 路径一律 `.trim()` 后使用
- tap 正则与 extract-code 正则分别优化，各自适配使用场景（流式 vs 整行匹配）
